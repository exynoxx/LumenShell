# Handoff — lumen-panel GTK4/GSK port: full migration + visual-parity restoration

**Created:** 2026-05-16 11:31 UTC
**Working directory:** /home/nicholas/Dokumenter/git/LumenShell
**Branch / commit:** master @ 6834ffc (working tree dirty — large unstaged changes from this session)

## Objective

Port `lumen-panel` from its custom GLES2 immediate-mode renderer
(`libdrawkit`) + custom Wayland client (`wlhooks`) to GTK4 +
`gtk4-layer-shell` + GSK while keeping the visual feel of the original
DrawKit panel. Then wire interactive behaviors (app entries, tray
expansion semantics, mouse-leave collapse, ESC, etc.) to match what
the old panel did.

## Context

- Prior feasibility work + foreign-toplevel PoC lived in
  `poc-gtk4-toplevel/` and the original `HANDOFF.md`. Both still on
  disk; the PoC remains as historical reference.
- The user is on KWin host so `zwlr_foreign_toplevel_manager_v1` is
  not advertised; nested Wayfire was the validation environment for
  the foreign-toplevel slice but visual checks during this session
  relied on the user running the binary themselves in their own
  Wayfire session.
- User decisions captured early in the session:
  - New code lives in `lumen-panel-gtk4/` during the port, then
    `lumen-panel/` (the drawkit-based original) is deleted and
    `lumen-panel-gtk4/` is renamed to `lumen-panel/` in a single
    cutover. **This rename has already happened** — current
    `lumen-panel/` is the GTK4 port.
  - `wlhooks` is extended (additively) to support GDK-owned
    `wl_display`; the legacy EGL/layer-shell/seat/output paths stay
    on disk but are no longer reached.
  - `libdrawkit` is no longer built. The directory was moved
    pre-session to `junkyard/libdrawkit/` (top-level `libdrawkit/`
    no longer exists). `DEPRECATED.md` was written into
    `junkyard/libdrawkit/` and `wlhooks/`.
  - Visual fidelity: pixel-near match to the original wherever
    feasible; otherwise the simpler GTK idiom wins.

### wlhooks additive surface

Three new symbols, **purely additive** — none of the existing
`wlhooks_init` / `init_layer_shell` / pointer/keyboard code paths
were modified beyond a single `if (!minimal_mode)` gate in `seat.c`:

- `wlhooks_init_toplevel_with_display(struct wl_display *)` in
  `wlhooks/wlhooks.h` / `wlhooks/main.c` — sets `minimal_mode`,
  inits only toplevel + activation + (minimal) seat, runs registry
  roundtrips on the external display.
- `wlhooks_destroy_toplevel()` — tears down only the above; does
  *not* call `wl_display_disconnect`.
- `seat_set_minimal_mode(bool)` in `wlhooks/protocols/seat.{c,h}` —
  suppresses pointer/keyboard attachment in the seat capability
  listener.

VAPI bindings added in `vapi/libwlhooks.vapi`:
`WLHooks.init_toplevel_with_display(Wl.Display)` and
`WLHooks.destroy_toplevel()`.

### App entries (left half)

- Launcher entry at index 0 (`app_id = "--"`), spawns `Utils.KICKOFF_BIN`.
- Per-app-id buttons appear when ToplevelStore announces a window.
- **Left-click semantics:**
  - Launcher → spawn Kickoff.
  - Entry with windows → cycles `ToplevelStore.activate` through
    its window_ids.
  - Pinned entry without windows → `launch_new_window()` spawns
    `launch_cmd` (from the `.desktop` Exec line).
- **Right-click (non-launcher only)** opens `AppPopupMenu`:
  - "Pin"/"Unpin" — toggles `is_pinned`. On unpin with no windows,
    `AppBar.wire_entry` removes the entry directly.
  - "New window" — `entry.launch_new_window()`. Dimmed when
    `launch_cmd == ""`.
  - "Close windows" — `entry.close_all_windows()`. Dimmed when
    no windows.
- Active window's entry draws a 5 px bottom underline in its
  `snapshot()` override (`AppEntry.snapshot`, fallback-coded RGBA
  since `@app_active_underline` from CSS doesn't resolve in code).
- Pinned-apps persistence: `~/.config/lumen-panel/pinned-apps.txt`
  via `Ini.read_lines/write_lines`.

### Tray (right half)

- Layout `[icon_row, revealer]` inside a `Gtk.Box(VERTICAL)` with
  `valign = END`, so when the revealer expands the icon row gets
  pushed upward — matches the original DrawKit behavior of a single
  rounded rectangle whose top edge grows up while the bottom stays
  anchored.
- CSS: `.tray-bar` is a floating rounded rect (all four corners,
  10 px radius, `margin-right: 20`, `margin-bottom: 6`); 
  `overflow = Gtk.Overflow.HIDDEN` is set in code so the page-slide
  animation can't bleed past the rounded silhouette.
- Icon order: WiFi, Battery, Sound, Clock, Exit. Page-bearing icons
  are the first three; Clock and Exit are leaf icons.
- `Gtk.Stack` with `SLIDE_LEFT_RIGHT` transitions handles
  page-to-page slides (direction follows insertion order).
- Active icon gets a `.active` CSS class (CSS mirrors `:hover`
  styling for that state).
- Auto-collapse via `Gtk.EventControllerMotion` attached to TrayBar,
  using `notify["contains-pointer"]` (not `leave` signal) because
  contains-pointer is documented as subtree-aware. 350 ms debounce
  is longer than the 280 ms reveal animation so a transient
  false-during-animation is harmless.
- ESC key on the window catches Escape and calls `tray.collapse()`.

### Services

Three `_service.vala` files are copied verbatim from the old tree
(battery sysfs reads; pactl polling at 1.5 s; nmcli background scan).
All three pre-warm their state in the constructor so the first tray
expand shows real data.

### Visual parity choices

- Battery + WiFi pages are **fully custom-drawn** (no `Gtk.LevelBar`,
  `Gtk.SearchEntry`, `Gtk.ListBox` chrome) because the user
  explicitly said GTK widgets weren't visually identical.
  `BatteryPage` is a single `Gtk.Widget` with a snapshot override
  drawing title / pct / status / progress / 2×2 stat grid in Pango.
  `WifiPage` uses `LumenChip`/`LumenTextField`/`LumenButton` (CSS
  class `.lumen-button`) + custom `WifiRow` snapshot widgets in a
  re-styled `Gtk.ScrolledWindow`.
- `SoundPage` still uses `Gtk.Scale` + `Gtk.ListBox` — **not yet
  converted** to custom drawing. User did not request it; preserving
  scope.

## Tried and rejected

- **`Gtk.EventControllerMotion.leave` signal on TrayBar.** Fires
  during the reveal animation as the layout shifts under the cursor
  and as the pointer crosses between TrayBar's internal children,
  causing the tray to collapse immediately on click. Replaced with
  `notify["contains-pointer"]` which is subtree-aware.
- **`[Revealer, icon_row]` layout for the tray.** Looked correct at
  first glance but put icons at the bottom of the expanded box with
  content above them — opposite of the original where icons are at
  the *top* of the expanded rectangle and page content fills the
  area below them. Current `[icon_row, revealer]` order with
  `valign = END` matches the original.
- **`Pango.attr_size_new(...)`.** Doesn't exist in the vapi; correct
  binding is `Pango.AttrSize.new_absolute(size_pt * Pango.SCALE)`.
- **`Gtk.Widget.size_allocate(Gtk.Allocation, baseline)`.** Vala vapi
  binds it as `size_allocate(int width, int height, int baseline)`.
  To position a child at an offset use
  `child.allocate(width, height, baseline, new Gsk.Transform().translate({x, y}))`.
- **`win.add_controller(esc)` for a `Gtk.EventControllerKey`.**
  Triggers an ambiguous overload error in Vala because
  `Gtk.ShortcutManager` also exposes an `add_controller`. Cast as
  `((Gtk.Widget) win).add_controller(esc)`.
- **CSS `color: white` to recolor SVG icons** that use
  `stroke="currentColor"`. Not honored unless the SVG is loaded as
  a GTK symbolic icon via icon theme. Worked around by editing
  `sound-max.svg` / `sound-mute.svg` to hard-code `stroke="#ffffff"`.
- **Vala `Object` ambiguity.** `using Json;` plus `class X : Object`
  is ambiguous between `GLib.Object` and `Json.Object`. Always
  spell `GLib.Object`.

## Open questions / blockers

- **No visual verification under Wayfire yet.** All testing this
  session was build + 2-second smoke-launch on KWin (no foreign-
  toplevel events, no real interaction). User needs to launch
  `./build/lumen-panel` in a Wayfire session and confirm:
  the floating tray reads correctly, the page-slide animation
  feels right, the AppEntry click semantics actually focus
  toplevels, the AppPopupMenu doesn't get clipped by the bottom
  layer-shell edge (sway issue #8518 risk), and the input region
  correctly passes clicks through the 6 px gap below the tray.
- **`Gtk.Overflow.HIDDEN` + rounded `border-radius`.** GTK4 docs
  say HIDDEN respects border-radius; not verified in this stack.
  If page content clips as a square at the corners during slide,
  fall back to a `snapshot()` override on TrayBar that pushes a
  rounded clip explicitly.
- **`SoundPage` still uses GTK widgets** (`Gtk.Scale`, `Gtk.ListBox`).
  Out of scope for this session. If visual mismatch is observed,
  apply the same treatment used on `WifiPage` /
  `BatteryPage` — needs new `LumenSlider` (snapshot widget with
  drag controller) and a custom sink-row widget.
- **GTK4 vulkan renderer regressions** on weaker GPUs. `env.sh`
  already defaults `GSK_RENDERER=ngl`; user can `unset` to test
  vulkan.
